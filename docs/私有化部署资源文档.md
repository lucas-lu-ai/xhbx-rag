# xhbx-rag 私有化部署资源文档

生成日期：2026-07-09

本文基于当前仓库代码、Docker Compose、环境变量样例和本机数据体量整理。当前项目本身不在应用容器内加载大模型权重，而是调用 OpenAI 兼容的 chat / embeddings / rerank HTTP 接口；因此主部署资源不需要 GPU。若要求模型也部署在私有环境，需要额外建设模型服务，见“全私有化模型资源补充”。

## 1. 部署结论

### 推荐部署形态

生产推荐使用 `docker-compose.yml` 的完整 Web 形态：

| 服务 | 作用 | 是否常驻 | 端口 |
| --- | --- | --- | --- |
| `web` | Nginx 托管 React 前端，并反代 `/api/` 到后端 | 是 | `WEB_PORT` 默认 `8080` |
| `api` | FastAPI / Uvicorn，提供问答、流式问答、批量执行、bad case、A2A 接口 | 是 | 容器 `8000`，宿主默认 `127.0.0.1:8000` |
| `standalone` | Milvus standalone 向量库 | 是 | `19530`，健康端口 `9091` |
| `etcd` | Milvus 元数据存储 | 是 | Compose 内网 |
| `minio` | Milvus 对象存储 | 是 | 调试端口 `9000/9001`，默认只绑定 `127.0.0.1` |
| `cli` | 按需执行解析、入库、检索调试等命令 | 否，`tools` profile | 无固定端口 |

如果只给 MCP 客户端暴露知识检索，不需要 Web UI，可以使用 `docker-compose.mcp.yml`：

| 服务 | 作用 | 是否常驻 | 端口 |
| --- | --- | --- | --- |
| `mcp` | MCP streamable-http 服务，默认暴露 `kb_search_knowledge` | 是 | `MCP_PORT` 默认 `9331`，路径 `/mcp` |
| `standalone` / `etcd` / `minio` | 同完整 Web 形态 | 是 | 同上 |

### 当前项目数据体量

当前工作区实际数据：

| 路径 | 当前大小 / 数量 | 部署意义 |
| --- | ---: | --- |

| `.local/` | 约 `53MB` | Web 批量 SQLite、bad case JSONL、本地调试文件 |

| 类型 | 当前模型 / 配置 |
| --- | --- |

| Web 批量并发 | 当前 `WEB_BATCH_CONCURRENCY=2` |

### 资源建议

不本地部署模型、只连接内网或专线模型 API 时：

| 场景 | CPU | 内存 | 磁盘 | GPU |
| --- | ---: | ---: | ---: | ---: |
| 当前体量最小可运行 | 4 vCPU | 16GB | 250GB SSD | 不需要 |
| 当前体量推荐生产规格 | 8 vCPU | 32GB | 500GB SSD / NVMe | 不需要 |
| Web + 频繁生成/入库 | 8-16 vCPU | 32-64GB | 500GB-1TB SSD / NVMe | 不需要 |
| MCP-only，且只保留 `parsed/` 和 Milvus | 4 vCPU | 8-16GB | 100-250GB SSD | 不需要 |

当前 `data/` 已有 153GB，完整 Web 私有化部署建议从 500GB 起步，避免后续新增素材、Milvus compaction、备份和 Docker 镜像把磁盘打满。

## 2. 服务清单

### 2.1 Web 前端 `web`

- 镜像来源：`web/Dockerfile`，Node 22 构建，Nginx 1.27 Alpine 运行。
- 前端技术栈：React 19、Vite、TypeScript。
- Nginx 行为：
  - 静态资源目录：`/usr/share/nginx/html`
  - `/api/` 反代到 `http://api:8000`
  - `client_max_body_size 50m`
  - 关闭 proxy buffering，支持后端 SSE 流式输出
  - `proxy_read_timeout` / `proxy_send_timeout` 为 3600 秒
- 资源：通常 0.25-0.5 vCPU、256-512MB 内存即可。

### 2.2 后端 API `api`

- 镜像来源：`Dockerfile.api`，Python 3.12 slim + uv。
- 启动命令：`uvicorn xhbx_rag.web.app:app --host 0.0.0.0 --port 8000`
- 主要能力：
  - `/api/status`：配置与索引状态检查
  - `/api/answer`：同步问答
  - `/api/answer/stream`：SSE 流式问答与步骤过程
  - `/api/source/reveal`：仅允许展示 `data/` 内引用源文件
  - `/api/bad-cases`：保存 bad case 到 `.local/bad_cases/bad_cases.jsonl`
  - `/api/batch-runs*`：批量问答会话，SQLite 持久化
  - `/a2a/xhbx-rag-answer`：A2A JSON-RPC 问答智能体
- 挂载目录：
  - `./data:/app/data`
  - `./.local:/app/.local`
  - `./generated:/app/generated`
  - `./parsed:/app/parsed`
  - `./scripts:/app/scripts:ro`
- 运行约束：
  - Web 批量执行依赖进程内后台线程，生产建议保持单个 Uvicorn worker。
  - `MILVUS_MODE=lite` 时本地 Milvus 文件只能串行使用；私有化生产建议使用 `MILVUS_MODE=docker`。
  - `WEB_BATCH_CONCURRENCY` 只在 `docker` Milvus 模式下允许大于 1，代码限制范围为 1-10。
- 资源：轻载 1-2 vCPU、2-4GB；批量并发或高并发建议 2-4 vCPU、4-8GB。

### 2.3 Milvus `standalone`

- 镜像：`milvusdb/milvus:v2.6.19`
- 模式：standalone
- 消息队列：`MQ_TYPE=woodpecker`
- 依赖：`etcd:2379`、`minio:9000`
- collection：
  - `MILVUS_COLLECTION=xhbx_sales_chunks`：绩优案例库
  - `MILVUS_COURSE_COLLECTION=xhbx_course_chunks`：培训课程库
- 当前 schema：
  - 主键：`chunk_id`
  - 向量字段：`FLOAT_VECTOR`
  - 度量：`COSINE`
  - 索引：`FLAT`
  - 文本、metadata、citations 以 VARCHAR / JSON 字符串形式保存
- 资源：
  - 当前 2631 chunk：2 vCPU、4GB 可运行。
  - 推荐生产：4 vCPU、8-16GB。
  - 超过 10 万 chunk 或高并发检索时，建议评估 ANN 索引或独立 Milvus 集群。

### 2.4 etcd

- 镜像：`quay.io/coreos/etcd:v3.5.25`
- 用途：Milvus 元数据。
- 数据卷：`milvus_etcd`
- 资源：1 vCPU、512MB-1GB 内存；磁盘预留 5-10GB。

### 2.5 MinIO

- 镜像：`minio/minio:RELEASE.2024-12-18T13-15-44Z`
- 用途：Milvus segment / object storage。
- 数据卷：`milvus_minio`
- 当前 compose 使用默认 `minioadmin/minioadmin`，生产必须改成私有强密码，或者只允许 Compose 内网访问。
- 资源：1 vCPU、1GB 内存；磁盘容量按 Milvus 数据与备份策略预留。

### 2.6 可选 MCP 服务 `mcp`

- Compose 文件：`docker-compose.mcp.yml`
- 镜像：同后端 API 镜像，命令为 `xhbx-rag-mcp --transport streamable-http --host 0.0.0.0 --port 9331 --path /mcp`
- 默认工具：`kb_search_knowledge(query, primaryDomains, knowledgeTypes=None, retrievalMode="HYBRID", hybridWeights=None, topK=10, includeDetails=false)`
- `primaryDomains` 必填，只允许产品知识、合规与风控、销售技能、客户经营、行业与公司、个人成长、组织发展。服务器问答智能体负责用自身大模型选择领域；MCP 只校验并转换为 `primary_domain` 硬过滤，不按 `source_kind` 区分培训资料与绩优案例。
- `kb_search_knowledge` 的标准 MCP 调用结果将业务对象放在 `CallToolResult.structuredContent`；
  新调用方应直接读取该字段，无需解析 `content[].text`。
  `content[].text` 仅作为旧客户端兼容回退保留。
- `includeDetails` 默认为 `false`：省略或设为 `false` 时，每条结果返回六字段精简切片；`docId` 取第一条 citation 的 `source_id`，缺失时为空字符串；`knowledgeType` 固定为 `SLICE`，`title` 固定为 `切片`，`content` 为完整正文，`primaryDomain` 和 `domainTags` 返回一级领域元数据：

  ```json
  {
    "docId": "pptx:案例A.pptx",
    "knowledgeType": "SLICE",
    "title": "切片",
    "content": "完整正文",
    "primaryDomain": "销售技能",
    "domainTags": ["销售技能", "客户经营"]
  }
  ```

  `includeDetails` 设为 `true` 时返回原完整切片结构。
- 当前 `kb_search_knowledge` 仅实现并返回 `SLICE`；`knowledgeTypes` 不包含 `SLICE` 时返回空 `data`，不会生成 `QA` 或 `KNOWLEDGE_POINT` 结果。
- `retrievalMode` 当前仅支持 `HYBRID`，传入 `SPARSE`、`VECTOR` 或其他值返回参数错误 `10004`。`hybridWeights` 仅为兼容占位参数：若提供则必须为对象，但其中权重当前不参与排序。
- 完整切片结构中的 `slice.citations` 是扩展字段，承载来源路径、文件名、定位信息和锚点。
- MCP 检索链不调用 chat/completions 做 query understanding 或领域判断，也不生成最终自然语言回答；只做参数校验、embedding、Milvus 混合召回、rerank。
- 服务器问答智能体提示词必须约束其调用前从七类体系选择 1 至 3 类，无法判断时传全部七类，不得省略 `primaryDomains`。
- `MCP_TOOL_PROFILE`：
  - `kb`：默认，只暴露 `kb_search_knowledge`
  - `legacy`：只暴露旧工具
  - `both`：新旧工具都暴露
- 资源：1 vCPU、1-2GB 内存；高并发可横向拆 MCP 与 Milvus。

### 2.7 可选 AgentScope Studio

- 当前项目可把 trace 发到 `WEB_STUDIO_ENDPOINT`，默认 `localhost:4317`。
- Studio 不在 compose 内，若生产启用需要额外部署 OTLP 接收端和 Web UI。
- 不启用时不影响问答与检索。

## 3. 模型调用清单

### 3.1 模型接口能力要求

| 模型类型 | 接口 | 当前用途 | 必需能力 |
| --- | --- | --- | --- |
| Chat 模型 | `/chat/completions` | 查询理解、答案生成、销售洞察生成、课程增值 | OpenAI 兼容；支持 JSON object 输出；生成流程需要 tool/function calling；流式回答需要 SSE |
| Embedding 模型 | `/embeddings` | query 向量化、chunk 入库向量化 | OpenAI 兼容 embeddings 返回；当前维度 4096 |
| Rerank 模型 | `/rerank` | 候选证据重排 | 请求包含 `model/query/documents`，响应包含 `results[].index` 与 `relevance_score` |
| Vision 模型 | DashScope / AgentScope 多模态调用 | 仅在 `VISION_MODEL_NAME` 非空时解析素材内嵌图片 | 支持 base64 图片输入，输出中文图像描述 |

### 3.2 在线问答调用路径

Web `/api/answer`、`/api/answer/stream` 和 A2A 问答通常每个问题调用：

1. Chat：`QueryUnderstandingAgent` 调用 `/chat/completions`，把原始问题改写成 `rewritten_query` 并抽取过滤条件。
2. Embedding：`EmbeddingClient.embed_query()` 调用 `/embeddings`，只向量化 `rewritten_query`。
3. Milvus：向量召回 topN。
4. 本地 BM25：从 Milvus 候选文本中做关键词召回。
5. RRF + 标签软加权：在后端内存中融合排序。
6. Rerank：`RerankClient.rerank()` 调用 `/rerank`，对候选文本重排到 topK。
7. Chat：`AnswerAgent` 以流式 `/chat/completions` 基于 topK 证据生成 JSON 回答和引用。

因此，单次有效问答的模型请求量约为：

| 入口 | Chat | Embedding | Rerank | 说明 |
| --- | ---: | ---: | ---: | --- |
| Web 同步/流式问答 | 2 次 | 1 次 | 1 次 | 问题不属于检索范围或无证据时可能跳过最终回答 Chat |
| A2A 问答 | 2 次 | 1 次 | 1 次 | 默认 `top_n=20`、`top_k=5` |
| CLI `search` | 1 次 | 1 次 | 1 次 | 只检索，不生成最终回答 |
| CLI `answer` | 2 次 | 1 次 | 1 次 | 同 Web 问答 |
| MCP `kb_search_knowledge` | 0 次 | 1 次 | 1 次 | 不做 query understanding，不生成自然语言回答 |

批量问答每一行等同一次 Web 问答，实际并发由 `WEB_BATCH_CONCURRENCY` 控制。

### 3.3 入库与生成调用路径

| 命令 | 模型调用 | 并发/规模口径 |
| --- | --- | --- |
| `generate-insights` | Chat 模型抽取章节级证据，再做案例级汇总 | 章节并发由 `--section-concurrency` 控制，默认 3；案例级默认 `--case-call-mode split`，按 4 类知识拆分调用 |
| `ingest` | `generate-insights` + parse + index | 生成阶段调用 Chat；入库阶段调用 Embedding |
| `parse-course` | 可选 Chat 模型生成课程摘要/受众/销售环节标签 | 默认开启；`--no-enrich` 可关闭 |
| `ingest-course` | `parse-course` + index | 课程增值可调用 Chat；入库阶段调用 Embedding |
| `index` | Embedding 模型批量向量化 chunks | 当前脚本对每个 `chunks.jsonl` 发起一次批量 embedding |

## 4. 存储容量估算

### 4.1 当前容量

完整保留当前项目数据，至少需要：

| 类型 | 当前占用 | 建议预留 |
| --- | ---: | ---: |
| 原始素材 `data/` | 153GB | 200GB 起 |
| 生成和解析产物 | 约 213MB | 10GB |
| Web 批量与 bad case `.local/` | 约 53MB | 10GB |
| Docker 镜像与 build cache | 依环境不同，当前可见 Milvus 镜像约 3.5GB | 20-50GB |
| Milvus 数据卷 | 取决于是否已入库与 compaction | 当前体量建议 20GB 起 |
| 日志与备份缓冲 | - | 50-200GB |

当前体量推荐 500GB SSD / NVMe。若只部署 MCP 检索，且不需要 Web 引用源文件，可不携带完整 `data/`，只携带 `parsed/` 入库产物和 Milvus 数据，磁盘可降到 100-250GB。

### 4.2 Milvus 向量容量公式

当前 embedding 维度为 4096，单条 raw vector 大约：

```text
4096 * 4 bytes = 16384 bytes ≈ 16KB
```

raw vector 容量估算：

| chunk 数 | raw vector 容量 | 建议 Milvus 磁盘预留 |
| ---: | ---: | ---: |
| 当前 2631 | 约 42MB | 20GB 起，主要给元数据、对象存储、日志和增长 |
| 10 万 | 约 1.53GB | 20-50GB |
| 100 万 | 约 15.26GB | 100GB+ |

实际 Milvus 占用还包括文本、metadata、citations、segment 文件、WAL/compaction 临时空间和备份。生产建议按 raw vector 的 2-3 倍以上预留，并额外保留系统和备份空间。

当前 Milvus schema 使用 `FLAT` + `COSINE`，检索精度高但计算随向量规模线性增长。若未来 chunk 数超过 10 万或需要较高 QPS，应评估改为 IVF/HNSW 等 ANN 索引或拆成独立 Milvus 集群。

## 7. 部署步骤

### 7.1 完整 Web 部署

1. 准备服务器目录，放置代码和数据目录：

```bash
cp .env.docker.example .env
```

2. 修改 `.env`：

- 填入 chat / embedding / rerank 模型服务地址与密钥。
- 设置 `MILVUS_MODE=docker`。
- 按模型限流和服务器能力设置 `WEB_BATCH_CONCURRENCY`，建议先从 2 或 3 开始。

3. 启动服务：

```bash
docker compose up -d --build
```

4. 查看健康状态：

```bash
docker compose ps
curl http://127.0.0.1:8000/api/status
```

5. 首次入库当前 `parsed/`：

```bash
docker compose --profile tools run --rm cli sh scripts/index_parsed.sh
```

首次希望清空旧 collection 再重建：

```bash
docker compose --profile tools run --rm -e RESET_COLLECTION=true cli sh scripts/index_parsed.sh
```

6. 浏览器访问：

```text
http://服务器IP:8080
```

### 7.2 MCP-only 部署

1. 生成离线包：

```bash
sh scripts/package_mcp_offline.sh
```

2. 在目标服务器解压后导入镜像：

## 8. 运维与安全建议

- 当前 Web API 和 MCP 服务自身没有用户鉴权；生产必须放在可信内网、VPN、零信任网关或 API 网关后面。
- MCP 服务 README 已提示：绑定 `0.0.0.0` 前必须确认处于可信内网，不要直接暴露公网。
- MinIO 默认账号密码必须改掉，或保持只在 Compose 内网可访问。
- 建议备份：
  - `data/`
  - `generated/`
  - `parsed/`
  - `.local/web_batch/batch_runs.sqlite3`
  - `.local/bad_cases/bad_cases.jsonl`
  - Docker volumes：`milvus_data`、`milvus_etcd`、`milvus_minio`
- Web 批量运行中断后，代码会把 running 状态标记为 interrupted，不会自动续跑，避免无人确认地产生模型费用。
- 生产日志需接入宿主机或平台日志采集，重点关注模型接口 429/5xx、Milvus health、磁盘水位和 batch 失败率。
- 如果使用外部模型 API，需要为后端容器配置稳定 DNS、代理或专线；模型接口超时和重试已经在代码里处理，但不能替代容量规划。

## 9. 全私有化模型资源补充

如果私有环境不允许访问 DashScope / SiliconFlow 等外部模型平台，需要另行部署模型服务，并把 `.env` 的三个 base URL 切换到内网地址。应用侧只要求接口兼容，不关心模型服务由 vLLM、TGI、LMDeploy、厂商推理平台还是其他框架承载。

### 9.1 必须部署的模型服务

| 服务 | 对应环境变量 | 当前模型 | 必需接口 |
| --- | --- | --- | --- |
| Chat / 生成模型服务 | `BASE_URL` / `MODEL_NAME` | 当前配置 `qwen3.7-max` | `/chat/completions` |
| Embedding 模型服务 | `EMBEDDING_BASE_URL` / `EMBEDDING_MODEL_NAME` | `Qwen/Qwen3-Embedding-8B` | `/embeddings` |
| Rerank 模型服务 | `RERANK_BASE_URL` / `RERANK_MODEL_NAME` | `Qwen/Qwen3-Reranker-8B` | `/rerank` |
| Vision 模型服务 | `VISION_MODEL_NAME` | 当前未启用 | 多模态 chat |

### 9.2 GPU 粗略口径

以下是模型服务侧的粗略资源口径，不属于本仓库 Docker Compose：

| 模型类型 | 粗略 GPU 建议 | 备注 |
| --- | --- | --- |
| Embedding 8B | 1 张 16-24GB VRAM GPU 起 | 取决于推理框架、精度、batch size 和并发 |
| Rerank 8B | 1 张 16-24GB VRAM GPU 起 | 可与 embedding 共享推理节点，但要压测排队延迟 |
| Chat 模型 | 取决于选型，通常是最大资源项 | 必须支持 JSON 输出、streaming、tool/function calling；若替换为 32B/72B 级模型，通常需要更高显存或多卡 |
| Vision 模型 | 仅启用图片理解时需要 | 当前 `VISION_MODEL_NAME` 为空，可不部署 |

如果模型也全私有化，建议把应用/Milvus 与模型推理拆成不同节点：

- 应用与 Milvus 节点：8 vCPU、32GB、500GB SSD 起。
- 模型节点：按模型规格单独规划 GPU、显存、推理并发和队列。
- 应用节点到模型节点之间保持低延迟内网链路，尤其是批量入库和批量问答场景。

## 10. 扩容判断

出现以下情况时，应从单机 compose 迁移到更强资源或拆分部署：

- `data/` 持续增长到 500GB 以上，备份窗口明显变长。
- chunk 数超过 10 万，`FLAT` 检索延迟不可接受。
- Web 批量并发长期达到 5-10，API CPU 或模型接口队列持续积压。
- Milvus standalone 内存持续高水位，compaction 或 load collection 变慢。
- MCP 与 Web 同时面向多个系统调用，需要独立限流和隔离。

优先扩容顺序：

1. 把模型服务独立为内网专用推理服务，应用只保留 API Key / endpoint。
2. 把 Milvus 从 compose standalone 独立到专用节点或集群。
3. 把 Web/API 与 MCP 拆成两个应用部署，分别限流。
4. 对大量历史数据评估 ANN 索引和 collection 分片策略。
