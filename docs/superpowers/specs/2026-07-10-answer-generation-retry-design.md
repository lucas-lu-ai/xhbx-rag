# 回答生成不完整时的纠错重试设计

## 背景

`AnswerAgent` 当前通过 OpenAI 兼容的 SSE 接口收集 `delta.content`，随后直接使用 `json.loads()` 和 `GeneratedAnswer` 校验结果。当模型输出被截断、流提前结束、SSE 数据块损坏，或模型返回了不符合约定的 JSON 时，代码只在最终解析处抛出异常。当前实现还没有检查 `[DONE]` 和 `finish_reason`，并且会静默忽略无法解析的 SSE 数据块，因此无法区分完整响应与半截响应，也不会让模型根据上一次错误自行纠正。

## 目标

- 模型输出不完整或结构不合法时，把上一次错误和不合法输出反馈给同一个模型，要求完整重新生成。
- 一次问答的模型内容生成总共最多尝试 3 次，包含首次生成。
- 任一次生成得到完整、符合模式的 JSON 后立即返回。
- 连续 3 次失败后停止重试，并向用户显示固定、安全的“模型输出不完整”提示。
- 保留现有网络连接级重试语义；网络重试与内容生成次数分别计数。

## 非目标

- 不对截断 JSON 进行本地补引号、补括号等修复。修复后的 JSON 可能语法有效但语义仍不完整，不应作为保险销售回答返回。
- 不修改检索、重排、引用选择或回答提示词的业务规则。
- 不把模型完整输出、检索证据、路径或密钥写入用户可见错误。

## 方案选择

采用“带上下文纠错重试”。相比无上下文重试，模型能看到具体失败原因并针对格式问题修正；相比本地 JSON 修复，该方案不会把被截断的语义误当作完整答案。

第一次生成使用当前系统消息和用户证据消息。解析或完整性校验失败后，下一次请求在原消息之后追加：

1. 一条 `assistant` 消息，包含上一次不合法输出的有界摘录；
2. 一条 `user` 纠错消息，包含错误类型、解析错误摘要、目标字段要求，以及“必须从头重新生成完整 JSON、不要续写上一次内容”的明确要求。

不合法输出摘录设置字符上限，只保留定位问题所需内容，避免三次请求造成上下文无界增长。每轮只反馈紧邻的上一次失败，不累计所有失败输出。

## 生成与校验流程

`AnswerAgent.generate()` 负责内容生成循环，最多执行 3 个逻辑生成尝试：

1. 构造本次消息并调用流读取函数。
2. 流读取函数收集正文、思考内容、是否收到 `[DONE]`、最后一个 `finish_reason`，并识别损坏的 SSE 数据块。
3. 按以下顺序严格校验：
   - SSE 数据块必须可解析；
   - 流必须收到 `[DONE]`；
   - `finish_reason` 不得为 `length`、`content_filter` 或其他明确的非正常结束原因；
   - 正文经过 Markdown JSON 围栏清理后必须可被 `json.loads()` 解析；
   - 解析结果必须通过 `GeneratedAnswer` 校验。
4. 校验成功时返回结果。
5. 校验失败且尚未达到第 3 次时，生成纠错消息并重新请求。
6. 第 3 次仍失败时，抛出专门的模型输出不完整异常，异常内部保留最后一次失败类型和摘要用于日志诊断。

正常兼容实现可能在 `[DONE]` 前提供 `finish_reason=stop`，也可能只提供 `[DONE]`。因此缺少 `finish_reason` 本身不判定失败；只有明确的异常原因才判定失败。

## 重试边界

内容生成尝试与现有传输重试分开：

- HTTP 状态码异常、连接建立失败等仍由 `_stream_chat_content()` 的传输重试处理。
- 已收到部分内容后发生连接中断、未收到 `[DONE]`、输出因长度停止、SSE 块损坏、JSON 解析失败或结构校验失败，均计为一次内容生成失败。
- 内容生成总次数固定为 3。即使底层连接级重试发生，也不会提高内容纠错循环的上限。

## 思考内容处理

每次尝试使用独立的正文和思考缓冲区，避免失败尝试的正文混入下一次。由于 Web 端会实时收到思考增量，已经发送的失败尝试思考无法撤回；重试开始时发送一条简短状态增量，说明上一次输出格式不完整、正在重新生成。最终响应的 `reasoning` 与界面已展示内容保持一致，按尝试顺序保留，并用清晰分隔符区分重试。

## 错误处理与安全性

模型纠错消息可以包含：

- 固定错误类别；
- `JSONDecodeError` 或模式校验的简短错误文本；
- 上一次模型输出的有界摘录。

用户可见错误必须使用固定文案，例如“模型输出不完整，已尝试 3 次，请稍后重试。”，不能直接透传解析异常、模型原文、文件路径或其他内部信息。

REST `/api/answer` 和 SSE `/api/answer/stream` 都通过共用的安全错误映射识别该专门异常：REST 返回 502 和固定文案；SSE 返回 `error` 事件和同一固定文案。服务端日志仍记录异常链，便于诊断。

## 可观测性

每次内容纠错重试记录不含原文的结构化信息：尝试序号、错误类别、正文字符数、是否收到 `[DONE]`、`finish_reason`。不记录 API 密钥、完整提示词、检索证据或完整模型输出。

## 测试策略

使用现有假 SSE 客户端覆盖以下行为：

- 第一次返回合法 JSON 时只调用一次。
- 第一次 JSON 不完整、第二次合法时成功返回，并验证第二次消息含错误反馈和上一次输出摘录。
- 前两次失败、第三次成功时总调用三次。
- 三次均失败时抛出专门异常，错误说明模型输出不完整且记录最后一次原因。
- 缺少 `[DONE]`、`finish_reason=length`、损坏 SSE 块分别触发内容纠错重试。
- 模式校验失败（缺少或清空 `answer`）触发内容纠错重试。
- 失败尝试的正文不会与成功尝试拼接。
- REST 和 SSE 对专门异常返回固定安全文案；其他未知异常继续返回通用兜底文案。
- 现有回答生成、引用映射、网络重试和 Web 流测试保持通过。

## 成功标准

- 短暂的模型格式错误能够在第 2 次或第 3 次生成时自动恢复。
- 单次问答的逻辑内容生成不会超过 3 次。
- 三次失败后，用户能明确看到模型输出不完整，而不会看到内部解析细节。
- 不完整、截断或本地修补的回答不会作为成功结果返回。
