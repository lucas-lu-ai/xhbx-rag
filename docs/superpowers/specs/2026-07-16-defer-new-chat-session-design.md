# 延迟创建新聊天会话设计

## 目标

点击 Web 界面的“新会话”后只进入空白草稿界面，不立即向历史会话列表或 `localStorage` 新增记录。用户首次发送非空问题时，才把草稿转成正式聊天会话。

## 范围

- 只改变用户主动点击“新会话”的行为。
- 页面首次打开时现有的默认空会话行为保持不变。
- 删除最后一个历史会话后现有的空白兜底会话行为保持不变。
- 批量会话、入库工作台和后端接口不变。

## 当前根因

`App.createSession()` 在按钮点击时调用 `createEmptySession()`，并立即把结果插入 `sessionStore.sessions`。`sessionStore` 的持久化副作用随后写入 `localStorage`，`SessionSidebar` 又直接渲染该数组，因此尚未发送问题的空会话会立即出现在列表中。

## 设计

### 临时草稿状态

`App` 单独维护一个只存在于 React 内存中的草稿会话。点击“新会话”时创建或替换该草稿，主区域显示空白 `ChatView`，但不修改 `sessionStore`，因此历史列表和持久化存储都不会变化。

草稿激活时，历史列表中不高亮任何旧聊天会话，避免主区域显示草稿但侧栏仍暗示旧会话处于选中状态。

### 首次发送时转正

`ChatView` 将“开始一轮问答”和后续流式更新区分开：

1. 用户提交非空问题。
2. `App` 检查提交目标是否为当前草稿。
3. 如果是草稿，使用首个问题生成标题和首个流式 turn，并一次性将会话插入 `sessionStore.sessions`，同时设为活动会话。
4. 草稿状态清空；后续思考、回答增量、完成或失败事件继续使用现有会话更新链路。

空白或纯空格提交不会创建会话。只要用户已经实际发送问题，即使回答请求失败，也保留正式会话和失败 turn，维持当前失败可追溯行为。

### 放弃草稿

- 草稿未发送前点击历史聊天、批量会话或批量执行时，直接丢弃草稿，不产生存储记录。
- 草稿未发送前再次点击“新会话”时，用新的空白草稿替换旧草稿，不累积空会话。
- 刷新页面时草稿自然丢失，因为它从未写入 `localStorage`。

## 组件边界

- `App.tsx`：拥有草稿生命周期、正式会话插入和选中态切换。
- `ChatView.tsx`：在首次提交时调用明确的“开始 turn”回调；流式事件仍调用现有更新回调。
- `chatSessions.ts`：继续负责正式会话的数据结构、标题和 turn 纯函数，不引入持久化草稿类型。
- `SessionSidebar.tsx`：仍只接收正式会话；通过草稿选中标识表现为历史列表无高亮项。

## 测试

在 `App.chat.test.tsx` 增加或调整集成测试，验证：

1. 已有历史会话时点击“新会话”，主区域变为空白，但历史列表条目数量和 `localStorage` 都不变。
2. 在草稿中首次发送非空问题后，历史列表新增一个以问题命名的会话，并写入 `localStorage`。
3. 未发送草稿后切回历史会话，不留下空会话记录。
4. 现有“首次打开默认空会话”和“删除最后会话生成兜底会话”测试继续通过。
5. 运行 Web 前端完整测试与构建，确认批量会话和其他界面契约未回归。
