# 模型自动选择检索 Collection 设计

## 目标

移除知识问答页面右上角的“索引状态”区域和手动 Collection 选择器。每次问答由查询理解模型根据问题内容选择案例知识库、课程知识库或同时选择两者，再只对选中的 Collection 执行检索。

单次问答、流式问答和批量执行共用同一套自动路由规则。

## 页面调整

- 删除右侧栏顶部整个“索引状态”卡片，包括状态、数据目录和 Collection 下拉选择器。
- “引用明细”区域上移并使用右侧栏的全部可用空间，不保留状态卡片造成的空白。
- Web 聊天请求不再提交 `collections` 字段。
- 状态接口仍用于读取检索参数和展示全局配置错误；移除状态卡片不影响错误横幅。
- 清理只服务于 Collection 选择器的前端状态、辅助函数、图标和样式。

## 模型路由契约

查询理解结果新增 `collection_targets` 字段，允许值为：

- `case`：案例经验、实战方法、客户旅程、异议处理和案例话术；
- `course`：培训课程、教材内容、标准流程和制式话术；
- `case` 与 `course`：问题同时需要两类知识，或无法可靠限定为单一知识库。

模型必须至少返回一个目标。解析层会去重并丢弃未知值；字段缺失、空数组或全部无效时，安全回退为 `case` 与 `course`，避免因为路由失败漏召回。

查询理解提示词明确要求：单一意图优先选择一个目标，混合意图选择两个，不确定时选择两个。现有 `chunk_types`、意图、问题改写和过滤条件继续保留，Collection 路由与知识类型过滤分别承担“查哪个库”和“库内查什么”的职责。

## 服务端数据流

问答服务按以下顺序执行：

1. 创建查询理解模型并对原始问题执行一次理解。
2. 将 `collection_targets` 映射到当前配置中的实际 Collection 名称。
3. 如果 API 调用方显式提供了 `collections`，继续将其视为兼容性覆盖；Web 页面不再提供该字段。
4. 创建只包含目标 Collection 的聚合检索视图。
5. 将已经得到的查询理解结果交给检索流水线，避免再次调用模型。
6. 继续执行 embedding、向量/关键词检索、重排和答案生成。

实际名称只由服务端配置提供，模型只输出稳定的 `case` / `course` 语义标识，避免把部署环境中的 Collection 名称写入提示词。

## 兼容性与异常处理

- `collections` API 字段暂时保留，保证已有 API 集成可以显式覆盖自动路由；Web 不再发送它。
- `search_evidence` 保留未传预计算查询理解结果时自行调用查询模型的能力，避免破坏 CLI 和已有单元测试调用方式。
- 自动路由只改变检索视图的 Collection 范围，不改变返回结构、引用索引、反馈数据或状态接口结构。
- 模型路由结果无效时检索两个已配置知识库；配置中缺少其中一个 Collection 时沿用现有聚合检索的降级行为。
- 查询理解调用失败继续按现有错误链路返回，不额外执行一次模型调用。

## 测试

- 查询理解模型测试覆盖 `case`、`course`、双目标、去重、未知值和缺失字段回退。
- 服务层测试覆盖模型选择案例库、课程库和双库时创建正确的检索视图，并确认查询理解模型只调用一次。
- 兼容性测试覆盖显式 `collections` 仍可覆盖模型路由。
- 前端测试覆盖“索引状态”和 Collection 选择器不再渲染，聊天请求不包含 `collections`。
- 布局测试或组件断言覆盖“引用明细”成为右侧栏首个且唯一的内容区。
- 运行相关前后端测试、完整前端测试、完整后端测试与生产构建。
