# 引用反馈双维度设计

## 目标

把右侧引用明细现有的“应该用/不该用”拆成两个可独立分析的质量维度：先判断引用召回是否准确，再在召回准确时判断模型回答是否正确参考该引用。每条引用形成一条完整反馈，避免把检索问题与生成问题混在同一个 judgement 中。

## 交互流程

每条引用采用逐级判断：

1. 首先显示“召回是否准确？”，选项为“召回准确”和“召回不准确”。
2. 选择“召回准确”后显示“回答是否正确参考该引用？”，选项为“参考正确”和“参考不正确”。
3. 选择“召回不准确”后不显示第二维度，回答参考判定记为“不适用”，并展开必填原因框。
4. 选择“参考不正确”后展开必填原因框。
5. 选择“召回准确 + 参考正确”时直接提交正向反馈。

两个维度都使用单选按钮。用户修改第一维度时，必须清空与新路径不兼容的第二维度和原因。反馈保存成功后沿用当前行为：锁定本条引用的反馈控件并显示成功提示。

负向原因框沿用现有样式，但根据路径显示不同的推荐填写内容：

- 召回不准确：“例如：该引用与客户问题无关、客户或案例不匹配，未能回答当前异议。”
- 参考不正确：“例如：回答曲解了引用原意、超出证据范围，或遗漏了关键限制。”

取消“召回不准确”的原因输入时清空第一维度；取消“参考不正确”的原因输入时保留“召回准确”，清空第二维度。

## 数据契约

前端每条新反馈写入以下字段：

```json
{
  "chunk_id": "case-a-1",
  "retrieval_judgement": "accurate",
  "answer_usage_judgement": "correct",
  "label": "案例A · 需求分析",
  "text_preview": "……",
  "reason": ""
}
```

枚举定义：

- `retrieval_judgement`: `accurate | inaccurate`
- `answer_usage_judgement`: `correct | incorrect | not_applicable`

组合约束：

- `accurate` 只能搭配 `correct` 或 `incorrect`。
- `inaccurate` 只能搭配 `not_applicable`。
- 两种负向组合必须携带非空 `reason`。

后端单问与批量接口继续兼容旧 `judgement` 数据，避免旧测试、历史调用或已有 JSONL 处理链突然失效；新前端只提交双维度字段。JSONL 落盘时为两个新枚举补充中文标签字段：

- `retrieval_judgement_label`: `召回准确 | 召回不准确`
- `answer_usage_judgement_label`: `参考正确 | 参考不正确 | 不适用`

## bad case 映射

| 用户选择 | feedback_result | issue_types | 原因 |
| --- | --- | --- | --- |
| 召回准确 + 参考正确 | `usable` | `usable` | 无 |
| 召回不准确 | `citation_issue` | `citation_wrong` | 必填 |
| 召回准确 + 参考不正确 | `inaccurate` | `answer_unsupported` | 必填 |

正向反馈继续不进入 bad case JSONL 导出；两类负向反馈进入导出。聊天与批量行仍复用同一份 payload 构建逻辑，批量模式继续走行级保存入口。

## 组件边界

- `EvidenceDetail` 负责两级单选、条件展示、原因输入和保存/取消状态。
- `BadCasePanel` 负责把终态判定转换成结构化 `EvidenceFeedback` 和 `BadCaseRequest`。
- `types.ts` 定义新枚举及反馈结构。
- FastAPI 单问与批量请求验证器接受并校验新组合，同时保留旧 judgement 兼容路径。
- `bad_cases.py` 只负责落盘标签补充，不改变 JSONL 追加写语义。

## 测试

前端组件测试覆盖：

- 初始只显示召回维度。
- 召回准确后显示回答参考维度。
- 两条负向路径分别显示正确的推荐 placeholder，原因为空不能保存。
- 切换第一维度会清理第二维度；取消原因按路径恢复正确状态。
- 正向组合直接提交，负向组合携带原因提交，保存后锁定。

集成测试覆盖聊天与批量两条提交路径的 payload 映射。后端测试覆盖合法组合、非法组合、负向原因必填、旧 judgement 兼容以及 JSONL 中文标签。

## 非目标

- 本次不增加回答级整体评价。
- 本次不评价未被模型实际引用的 top-k 候选。
- 本次不改变检索、回答生成或批量任务状态机。
